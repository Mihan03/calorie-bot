# Telegram-бот подсчёта калорий — Архитектура MVP

> Документ фиксирует решения, к которым мы пришли в ходе обсуждения.
> Главный принцип проекта: **каждая технология должна быть обоснована задачей**, а не желанием «попробовать». Если технология не решает конкретную проблему — она отложена в backlog, а не добавлена ради галочки.

---

## 1. Суть продукта

Пользователь присылает в Telegram фото еды (можно несколько за раз). Система распознаёт блюда и КБЖУ через AI (Vision-модель), сохраняет приёмы пищи, считает прогресс относительно дневной цели и даёт краткие советы. Доступна статистика за день / неделю / месяц и генерация отчёта.

**Заложено на будущее (НЕ в MVP):** данные сна из Apple Watch, замеры InBody из DDX.

---

## 2. Ключевое архитектурное решение

Не «чистые микросервисы» и не «монолит», а **модульный монолит с сателлитами**: основное приложение с чёткими границами модулей, вокруг — несколько специализированных сервисов, общающихся через Kafka и REST.

**Почему так, а не «всё в микросервисы»:** при небольшой нагрузке и одном разработчике полноценные микросервисы — оверинжиниринг. Сервисы выносятся только там, где для этого есть техническая причина, а не «для резюме».

### Критерии выноса в отдельный сервис
1. **Разный профиль нагрузки** — кусок потребляет ресурсы иначе, чем остальное.
2. **Асинхронность** — операция долгая, её нельзя делать в синхронном запросе.
3. **Изоляция отказа** — падение этого куска не должно ронять остальное.

### Что вынесено и почему
| Сервис | Причина выноса |
|--------|----------------|
| `ai-service` | Самый сложный модуль: платные вызовы, нестабильность внешнего API, долгая операция. Изоляция отказа + асинхронность. |
| `report-service` | Долгая операция (генерация отчёта). Асинхронность. |

## 3. Сервисы

| Сервис | Зона ответственности | Тип | Хранилище |
|--------|---------------------|-----|-----------|
| **gateway** | Единая входная дверь. Принимает апдейты Telegram, проверяет `secret token`, достаёт `userId`, маршрутизирует дальше. Тонкий, бизнес-данными НЕ владеет. | Транспорт (вход) | — (stateless) |
| **user-service** | Владелец профиля и целей КБЖУ. Регистрация, onboarding, FSM-состояние диалога. Owner-check. | Домен | PostgreSQL |
| **intake-service** | Владелец приёмов пищи (что съел, КБЖУ, когда). Запись результатов AI, удаление, агрегация дневного итога. | Домен | PostgreSQL + Redis (дневной кэш) |
| **ai-service** | Распознавание еды и КБЖУ по фото через Vision-модель. Управление промптами. | Домен (обработка) | — (stateless, читает фото из события) |
| **report-service** | Генерация отчётов день/неделя/месяц из данных intake. Агрегатор-читатель. | Домен (чтение) | — (генерация на лету; файлы → объектное хранилище при необходимости) |
| **notification** | Единая исходящая дверь. Слушает события о *свершившихся фактах*, отправляет сообщения в Telegram API. | Транспорт (выход) | — |

> **Важно про две «двери»:** `gateway` принимает входящие, `notification` отправляет исходящие. Это разные пути. Notification всегда стоит **в конце** цепочки — уведомлять можно только о факте, который уже стал правдой в системе.

## 4. Аутентификация и авторизация — два разных слоя

| Что | Где живёт | Механизм |
|-----|-----------|----------|
| «Запрос реально от Telegram?» (аутентификация входа) | gateway | Проверка заголовка `X-Telegram-Bot-Api-Secret-Token` (задаётся при установке webhook) |
| «Этот userId смотрит свой отчёт, а не чужой?» (owner-check / row-level) | домен (user/intake) | Бизнес-правило по данным. **Не** работа Spring Security |

**Различие, важное для собеседований:** Spring Security охраняет *двери и роли* (кто ты, какая роль). Принадлежность *конкретных данных* конкретному пользователю охраняет домен — фреймворк не знает, что отчёт №123 принадлежит юзеру №45.

## 5. Взаимодействие: REST vs Kafka

**Правило:** REST — для команд/запросов, где клиент ждёт ответа *сейчас*. Kafka — для асинхронных фактов-событий. **Не всё подряд через Kafka.**

### Синхронный REST
- Задать цель КБЖУ → `gateway → user-service`
- `/start` и регистрация → `gateway → user-service`
- «Остаток на сегодня» (кнопка) → `gateway → intake-service`
- Удалить запись о еде → `gateway → intake-service` *(владелец еды — intake, не user!)*
- Запросить отчёт → `gateway → report-service` (мгновенный ack «готовлю», НЕ синхронная генерация)

### Асинхронные события (Kafka)
- Загружены фото еды
- AI закончил распознавание
- Еда сохранена
- Отчёт готов

> **Паттерн долгой операции:** быстрый синхронный ack («принял, считаю») + асинхронная доставка результата событием. Применяется и к фото, и к отчётам. Закладываемся на медленный путь, чтобы не упереться в таймауты.

## 6. Поток обработки фото (центральный сценарий)

```
1. Пользователь шлёт 10 фото
        │
        ▼
2. gateway: проверил secret token, достал userId
        │  publish: PhotoUploaded
        ▼
3. [Kafka] топик photo.uploaded
        │
        ▼
4. ai-service (consumer): фото → Vision-модель → КБЖУ
        │  publish: IntakeProcessed
        ▼
5. [Kafka] топик intake.processed
        │
        ▼
6. intake-service (consumer): пишет приём пищи в БД
        │  (в ТОЙ ЖЕ транзакции пишет в outbox)
        │  publish: IntakeSaved  ← через outbox-publisher
        ▼
7. [Kafka] топик intake.saved
        │
        ▼
8. notification (consumer): шлёт пользователю
   «добавил, осталось X до нормы»
```

**Ключевое решение (consistency):** notification слушает `IntakeSaved`, а **не** `IntakeProcessed`. То есть уведомление уходит только *после* реальной записи в БД. Иначе возможен рассинхрон: AI посчитал, notification написал «всё ок», а запись в БД упала — пользователь думает, что еда учтена, а её нет.

## 7. Transactional Outbox + идемпотентность

### Проблема (dual write)
intake-service должен сделать две вещи: записать в БД **и** опубликовать событие в Kafka. Общей транзакции на БД + Kafka не существует. Если БД записалась, а publish упал — рассинхрон.

### Решение: Transactional Outbox
- Событие пишется в таблицу `outbox` **той же транзакцией**, что и приём пищи → атомарность гарантирована БД.
- Отдельный publisher вычитывает `outbox` и публикует в Kafka.
- **MVP-вариант publisher'а:** фоновый `@Scheduled`-job, опрос таблицы. *(Debezium/CDC — next step, не в MVP.)*

### Подводные камни (важно!)
- Outbox даёт **at-least-once**: publisher может опубликовать событие повторно (упал после publish, до пометки «отправлено»). Outbox **не** убирает дубли — он перекладывает их на consumer.
- Поэтому **consumer обязан быть идемпотентным**: каждое событие несёт уникальный ключ (`update_id` Telegram или сгенерированный id), consumer проверяет «уже обрабатывал?» перед записью.
- **Outbox имеет смысл только в паре с идемпотентным consumer'ом** — это две стороны одной монеты.

> Дополнительный источник дублей: Telegram может прислать апдейт дважды. Тот же механизм идемпотентности по `update_id` это закрывает.

## 8. Хранилища данных

**Правило:** одна БД на сервис, общей БД нет (иначе это распределённый монолит — худшее из миров). «Отдельная БД» ≠ «отдельный инстанс» — допустимы разные схемы/логические БД в одной СУБД при малой нагрузке.

| Данные | Где | Обоснование |
|--------|-----|-------------|
| Профиль + цели КБЖУ | user / PostgreSQL | Долговечные бизнес-данные |
| **Состояние диалога onboarding** (ввод цели, регистрация) | user / **PostgreSQL** | Это процесс user-сервиса → его состояние. Потеря шага onboarding ощутима → надёжность оправдана |
| **Состояние диалога истории** (показал список → жду номер, маппинг номер→uuid) | intake / **PostgreSQL** | Это данные и процесс intake. Потеря почти не критична (один лишний тап), но кладём в Postgres ради простоты |
| Приёмы пищи | intake / PostgreSQL | Бизнес-ядро |
| **Дневной кэш «остаток на сегодня»** | intake / **Redis** | Частый запрос (кнопка), пересчёт суммы за день по Postgres дорого. Инвалидация при новом приёме пищи. |
| Отчёты | report / — | Агрегатор, генерация на лету |

> **Важный принцип (выведен в обсуждении):** состояние диалога НЕ принадлежит одному сервису целиком — оно распадается по доменам. Каждый сервис владеет состоянием *своих* диалогов, никто не лезет в чужую БД. Критерий выбора хранилища — не ярлык «важно/неважно», а **что конкретно сломается при потере этих данных**.

**Миграции:** **Liquibase** — обязательно. Миграции живут **внутри каждого доменного сервиса** (`user-service`, `intake-service`), рядом с его кодом — НЕ в отдельном общем модуле. Вынос миграций в общий модуль неявно связал бы схемы разных сервисов в одном месте = общая БД через чёрный ход, что ломает изоляцию.

## 10. Структура монорепозитория (Gradle multi-module)

Один git-репозиторий, Gradle multi-module build. **Монорепо ≠ монолит:** репозиторий — про код, сервис — про рантайм. Все шесть сервисов автономны в рантайме уже сейчас, просто их исходники лежат вместе.

### Дерево модулей
```
calorie-bot/
├── settings.gradle              # подключение всех модулей
├── build.gradle                 # общая конфигурация
├── gateway/                     # сервис (входная дверь)
├── user-service/                # сервис + своя БД (Liquibase внутри)
├── intake-service/              # сервис + своя БД (Liquibase внутри)
├── ai-service/                  # сервис
├── report-service/              # сервис
├── notification/                # сервис + processed_messages (своя БД)
├── events/                      # LIBRARY: классы событий Kafka (контракт)
└── common/                      # LIBRARY: логирование, trace_id, общие утилиты (минимальный!)
```

### Различие модулей
- **`*-service`, `gateway`, `notification`** — деплоятся, запускаются, имеют `main`, свой жизненный цикл.
- **`events`, `common`** — library-модули, НЕ запускаются, подключаются как зависимость. У них нет «инстанса», падать в рантайме нечему — они вкомпиливаются в каждый сервис на этапе сборки.

### Раскладка общего кода
| Модуль | Что внутри | Почему |
|--------|-----------|--------|
| `events` (library) | Только классы событий Kafka (`PhotoUploaded`, `IntakeProcessed`, `IntakeSaved`, `ReportReady`) | Контракт между сервисами. Общий, т.к. producer и consumer десериализуют один байтовый поток — расхождение классов ломает десериализацию молча |
| `common` (library) | Конфиг логирования, trace_id, базовые исключения | Держать **минимальным**: жирный common = связанность по сборке (изменил → пересобрать всех) |
| REST-DTO | **Локально в каждом сервисе** (свои request/response) | Осознанный trade-off: дублирование — цена за независимость сервисов. DRY между сервисами = антипаттерн (создаёт связанность). |
| Liquibase-миграции | **Внутри каждого доменного сервиса** | Изоляция схем БД |

### Подключение `events` и его цена
- **Пока всё в монорепо:** `implementation project(':events')` — project dependency, версий нет, всё синхронно. ← это MVP.
- **Когда library уезжает в отдельный репо:** публикация в artifact registry (Nexus/Artifactory/GitHub Packages) + зависимость по версии `com.bot:events:1.2.0`.
- **Скрытая цена общего `events`:** он — *якорь*. Как только хоть один потребитель оказывается вне репозитория с `events` — общий код обязан стать опубликованным артефактом, **и тогда ВСЕ потребители разом** переходят с `project()` на версионную зависимость. Это сдвиг для всех, не для одного.

> Выбор по событиям: **вариант A** (общий `events`-модуль) — для MVP проще. Альтернатива (schema registry, Avro/Protobuf, генерация классов у каждого) — overkill для MVP, но знать стоит.

## 11. Схема базы данных

> Общее: у всех таблиц **UUID как PK** (не внешний tg_id — чтобы данные не висели на чужом идентификаторе). Базовый абстрактный класс (`@MappedSuperclass`) даёт `id / created_at / updated_at` всем сущностям.

### 11.1 user-service

**`users`**
| Колонка | Тип | Примечание |
|---------|-----|------------|
| id | UUID | PK (базовый класс) |
| created_at, updated_at | TIMESTAMP | базовый класс. `created_at` = момент регистрации (отдельное `date_registry` НЕ нужно — это дубль) |
| tg_id | BIGINT | **NOT NULL, UNIQUE** — связь с Telegram, без него юзера не найти по апдейту |

**`goals`** — история целей (one-to-many: user → goals)
| Колонка | Тип | Примечание |
|---------|-----|------------|
| id | UUID | PK |
| user_id | UUID | FK → users |
| calories, protein, fat, carbs | INTEGER | КБЖУ (целые; **не FLOAT** — копит ошибку округления при суммировании) |
| created_at | TIMESTAMP | = момент начала действия цели |
| valid_to | TIMESTAMP, nullable | когда перестала действовать. **`NULL` = активная сейчас** |
| updated_at | TIMESTAMP | базовый класс |

- Активная цель: `WHERE user_id = ? AND valid_to IS NULL`
- Цель на дату X: `WHERE user_id = ? AND created_at <= X AND (valid_to > X OR valid_to IS NULL)`
- **Зачем история:** при смене цели старая статистика (проценты от нормы) не должна пересчитываться задним числом. Принцип: исторические данные не зависят от мутабельного текущего значения.
- **Защита инварианта «одна активная цель»:** partial unique index (обычный UNIQUE не подходит — NULL'ы не сравниваются):
```sql
CREATE UNIQUE INDEX idx_one_active_goal_per_user
ON goals (user_id) WHERE valid_to IS NULL;
```
- Смена цели = в **одной транзакции**: закрыть старую (`valid_to = now()`) + вставить новую.

**Состояние диалога onboarding** — отдельная таблица (НЕ колонка в users): FSM-шаг регистрации.

### 11.2 intake-service

**`intakes`** — приёмы пищи
| Колонка | Тип | Примечание |
|---------|-----|------------|
| id | UUID | PK |
| created_at, updated_at | TIMESTAMP | базовый класс. `created_at` ещё и для **сортировки/нумерации** истории |
| user_id | UUID | чей приём пищи |
| date | DATE | бизнес-дата (выбрана кнопкой), для группировки по дням |
| name_dish | VARCHAR | название блюда (одно фото → N блюд по ответу нейронки) |
| calories, protein, fat, carbs | INTEGER | КБЖУ |
| ai_comment | VARCHAR | пояснение от нейронки |
| user_comment | VARCHAR, nullable | заметка пользователя (для отладки) — отдельно от ai_comment, источники не мешаем |
| deleted_at | TIMESTAMP, nullable | **soft delete**: активные = `WHERE deleted_at IS NULL`. История не теряется |

- Редактирование = **UPDATE** записи (не delete+insert). Получаем новые данные от AI, находим по id, обновляем.
- **Порядковый номер блюда — НЕ колонка.** Вычисляется при выводе истории (сортировка по `created_at`, нумерация 1..N на лету). Номер в БД нестабилен при удалении. Маппинг «показанный номер → uuid» хранится в состоянии диалога.
- **UX-приём (на этапе бота):** удаление через кнопку с uuid в `callback_data` (не ввод номера текстом); длинные списки — пагинация по 5.

**`outbox`** — transactional outbox (producer-side)
| Колонка | Тип | Примечание |
|---------|-----|------------|
| id | UUID | PK строки outbox |
| created_at, updated_at | TIMESTAMP | базовый класс |
| message_key | UUID/VARCHAR | **NOT NULL, UNIQUE** — ключ идемпотентности, едет в Kafka. **На каждое блюдо** (одно фото → 2 блюда → 2 ключа), стабилен между повторами |
| event_type | VARCHAR | NOT NULL, тип события (`IntakeSaved`), неизменный |
| payload | JSONB | NOT NULL, **тело события целиком**, готовое к отправке (publisher не лезет обратно в intakes) |
| aggregate_id | UUID | ссылка на приём пищи, для трассировки |
| status | enum | `PENDING → SENT`, publisher меняет после публикации |
| sequence_no | BIGSERIAL | (опц.) строгий порядок, если `created_at` мало |

- Пишется в **той же транзакции**, что и приём пищи → атомарность.
- Publisher (`@Scheduled`) читает PENDING по порядку → публикует в Kafka → ставит SENT.
- **Дубль при сбое между publish и SENT — допустим и ожидаем**, гасится идемпотентным consumer'ом.

**`processed_messages`** — дедупликация (consumer-side, в notification и любом consumer'е)
| Колонка | Тип | Примечание |
|---------|-----|------------|
| message_key | UUID/VARCHAR | **PK / UNIQUE NOT NULL** — по нему дедуп |
| created_at | TIMESTAMP | для последующей чистки старых записей (таблица растёт вечно) |

- Тощая таблица — только ключ + время. **НЕ** payload (он уже в Kafka-событии), не event_type. Чем тоще — тем быстрее проверка (она на каждом сообщении).
- Дедуп атомарно: `INSERT ... ON CONFLICT (message_key) DO NOTHING` — НЕ `SELECT`-then-`INSERT` (race condition: два дубля параллельно оба увидят «пусто» и оба обработают).
- Вставка ключа + бизнес-эффект — в **одной транзакции**.
- **Где живёт:** у каждого consumer'а своя (notification не может лезть в outbox чужого сервиса).

### Гранулярность ключа идемпотентности
Ключ переопределяется на границе, где меняется гранулярность операции:
- **Уровень приёма фото** (gateway→ai): ключ от `update_id` — чтобы не прогнать одни фото через платный AI дважды.
- **Уровень сохранённого блюда** (intake→notification): одно фото → N блюд → N ключов. Иначе при «один ключ на два блюда» второе блюдо сочтётся дублем и потеряется.

## 12. Логирование и наблюдаемость

> **Терминологическая рамка:** логирование — **не сервис**. Это библиотека, вкомпилированная в каждое приложение. Отдельным процессом крутится только *сборщик* логов (Loki/ELK), но его не пишут — берут готовый. Конфиг логирования и MDC-утилиты живут в library-модуле **`common`**.

### Базис (уже в стеке, выбирать не надо)
- **SLF4J** — абстракция, пишем только через неё (`@Slf4j` от Lombok, `log.info(...)`). **Никогда** не привязываться напрямую к `java.util.logging` или Log4j API.
- **Logback** — реализация, идёт из коробки со Spring Boot.
- **Структурированное (JSON) логирование встроено в Spring Boot начиная с 3.4** → в нашем Boot 4.x это конфигурация, а не сторонняя библиотека.

### Зачем JSON-логи
Текстовый лог нельзя нормально искать. Структурированный = каждая строка это JSON с полями: `level`, `message`, `service`, `traceId`, `userId`. Тогда:
- фильтр по `traceId` → все строки одного запроса по всем сервисам;
- алерт по `level:ERROR` без парсинга текста;
- агрегация по логгеру → видно, кто шумит.

### Correlation ID через MDC — ядро для микросервисов
**MDC (Mapped Diagnostic Context)** — thread-local карта, содержимое которой Logback автоматически подставляет в каждую строку лога. Кладёшь ID один раз на входе — он попадает во все логи, без протаскивания параметром через каждый метод.

**Стандартная механика:** ID генерируется в точке входа → пропагируется через HTTP-заголовок `X-Correlation-ID` (де-факто стандарт) → если сверху не пришёл, генерируется `UUID.randomUUID()`.

**Наш нюанс:** типовой рецепт «сервлет-фильтр на входе» в gateway **не работает** — при long polling входящего HTTP нет, апдейты приходят изнутри библиотеки. Поэтому:

| Сервис | Где рождается / читается correlation ID |
|--------|------------------------------------------|
| **gateway** | В начале `consume(Update)` — единственный метод, вызываемый ровно раз на апдейт |
| **user-service и прочие** | Сервлет-фильтр читает заголовок `X-Correlation-ID` (здесь уже настоящий HTTP) |

**Шаблон в gateway:**
```java
consume(Update update) {
    MDC.put("correlationId", UUID.randomUUID().toString());
    try {
        ... вся обработка апдейта ...
    } finally {
        MDC.clear();
    }
}
```

- ID кладётся **в самом начале**, до обработки — иначе логи до вызова user-service (и логи ошибок, случившихся раньше вызова) останутся без ID, а это как раз самое интересное при разборе сбоя.
- **`try/finally` + `MDC.clear()` обязателен.** MDC в thread-local, потоки в пуле переиспользуются. Не почистил → следующий апдейт на том же потоке унаследует чужой ID, логи слипнутся.
- При вызове другого сервиса ID достаётся из MDC и кладётся в заголовок `X-Correlation-ID`.

### Ограничение MDC (важно для нашей архитектуры)
**MDC живёт в thread-local и через асинхронные границы сам не переезжает** — при переходе в другой поток или через очередь контекст надо пробрасывать вручную. У нас вся архитектура на Kafka → асинхронных границ много. Взрослое решение — **Micrometer Tracing / OpenTelemetry**, где `traceId`/`spanId` согласованно сохраняются через границы сервисов, очереди и async-потоки.

### Порядок внедрения (не всё сразу)
| Этап | Что делаем |
|------|-----------|
| **1. Сейчас** | Модуль `common`: JSON-логи + константы имён MDC-ключей. MDC в `consume()` gateway |
| **2. С появлением user-service** | Пропагация ID через заголовок `X-Correlation-ID` + сервлет-фильтр на принимающей стороне |
| **3. С появлением Kafka** | Micrometer Tracing — MDC через очередь не переедет, упрёмся на практике |
| **4. При 3+ сервисах** | Агрегатор логов (Loki/ELK) + Grafana. Не раньше — пока хватает консоли |

### Прочее
- **Логировать осмысленно** — не всё заслуживает INFO.
- Что класть в MDC кроме correlation ID: `tgId` пользователя (плюс — фильтр всех логов по конкретному юзеру при разборе жалобы; минус — персональные данные в логах, для tg_id некритично, но привычку осознавать стоит завести).

## 13. Известные риски / продуктовые решения

- **Точность распознавания КБЖУ по фото нестабильна.** Заложить подтверждение/правку распознанного пользователем.
- **AI Vision платный.** Нужны лимиты на пользователя (rate limiting — кандидат на Redis) и понимание, кто платит.
- **Onboarding-состояния.** Пользователь может нажать `/start` и пропасть. Нужны промежуточные состояния пользователя; присланное до завершения onboarding фото обрабатывается по правилам FSM.
- **Наблюдаемость.** Spring Boot Actuator + Prometheus/Grafana, structured logging с trace_id через сервисы — для распределёнки обязательно (можно поэтапно).
